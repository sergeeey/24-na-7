# Incident Memory System

**Назначение:** не гасить симптомы, а накапливать причинно-следственную модель поломок и закрывать цикл «ошибка → защита от повтора».

- **Реестр:** [ledger.yaml](./ledger.yaml) — одна запись на повторяемый инцидент.
- **Правило закрытия:** баг не считается закрытым, пока нет **хотя бы одного** из:
  1. автоматический (регрессионный) тест,
  2. runtime guard (алерт, проверка, отказ от опасного пути),
  3. запись в incident ledger с **проверяемым signpost** (что будем смотреть, чтобы убедиться, что не повторилось).

---

## Операционный ритуал после инцидента (≤10 мин)

Проводится после каждого повтора; ответы фиксируются в карточке в `ledger.yaml`.

1. **Что сломалось?** — одна фраза (симптом на уровне пользователя/системы).
2. **Какой был первый наблюдаемый симптом?** — что первым увидели в логах/UI/метриках.
3. **Что было настоящей причиной?** — корневой механизм, а не «опять 404».
4. **Какой сигнал мы должны были увидеть раньше?** — signpost для мониторинга/логов.
5. **Какой guard теперь не даст этому повториться?** — тест, guardrail или явный мониторинг signpost.

После ритуала: обновить карточку в ledger (root_cause, guardrail, signpost, regression_test, status).

---

## Поля карточки инцидента

| Поле | Описание |
|------|----------|
| `incident_id` | Уникальный ID (INC-NNN). |
| `signature` | Короткая уникальная формулировка для поиска и тегов. |
| `symptoms` | Наблюдаемые проявления. |
| `root_cause` | Установленная или гипотетическая первопричина. |
| `evidence` | Где смотреть (логи, файлы, метрики). |
| `what_worked` | Что помогло при обходе/фиксе. |
| `what_failed` | Что не сработало или скрывало причину. |
| `guardrail` | Защита от повтора (код, конфиг, правило). |
| `regression_test` | Автотест или сценарий проверки. |
| `signpost` | Конкретный индикатор для мониторинга/логов. |
| `owner` | Ответственный за доведение до закрытия. |
| `status` | open / in_progress / closed. |

## Зачем это нужно

- **Observability есть, debugging есть, institutional learning не было.**  
  Система чинилась локально, но не «запоминала» причину и не ставила guard.
- Сложная нелинейная цепочка (mobile → websocket → ingest → pipeline → enrichment → storage) даёт повторяющиеся сбои из петель обратной связи и задержек. Без явной каузальной модели команда спорит о симптомах, а не о слое, где ломается.
- Fallback и обходные пути скрывают правду: в debug нельзя молча уходить на remote — иначе эксперимент невоспроизводим.
- Incident Memory встраивается в процесс: каждый повтор → ритуал → обновление ledger → guard/signpost. Так накопление инженерного знания становится обязательным контуром, а не идеей.
