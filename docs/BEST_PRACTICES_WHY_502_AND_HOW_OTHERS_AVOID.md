# Почему у тебя 502, а у других нет: лучшие практики

Краткий разбор: **почему** возникают 502 и рост очереди, **как** устроены мировые практики и **что** можно сделать в Reflexio.

---

## 1. В чём корневая причина

### Что происходит у тебя сейчас

```
Телефон отправляет WAV по WebSocket
        ↓
Сервер принимает байты и в той же связи запускает:
  → фильтр речи
  → Whisper (ASR)     ← 20–90+ сек на сегмент
  → enrichment (LLM)  ← ещё секунды
  → запись в БД
        ↓
Только после всего этого сервер шлёт "received" и "transcription"
        ↓
Прокси (nginx/cloud) ждёт ответ от backend. Дефолт — 60 сек.
        ↓
Если обработка дольше → proxy_read_timeout → 502 Bad Gateway
```

То есть **тяжёлая работа выполняется внутри одного запроса/соединения**. Прокси не видит ответа вовремя → обрывает соединение → 502. Клиент не получает ack → не удаляет запись из очереди → очередь растёт.

### Почему у «других» так не ломается

В продакшн-системах обычно делают иначе:

- **Не держат** долгую обработку в HTTP/WS-обработчике.
- **Сразу** принимают данные, отдают клиенту «принято» и закрывают быстрый цикл.
- **Обработку** (STT, NLU, обогащение) выносят в **очереди и воркеры** (Celery, SQS, Lambda, K8s jobs и т.п.).

Тогда:

- Ответ клиенту — за миллисекунды/секунды.
- Таймауты прокси не упираются в длительность транскрипции.
- 502 из-за «долгого ответа» не возникает.

## 2. Мировые практики (кратко)

### 2.1. Сервер: «принял → ответил → обработал потом»

Источники: [AWS Prescriptive Guidance](https://docs.aws.amazon.com/prescriptive-guidance/latest/patterns/process-events-asynchronously-with-amazon-api-gateway-amazon-sqs-and-aws-fargate.html), [Medium: API Gateway + queue](https://medium.com/@umutt.akbulut/api-gateway-is-up-but-the-system-is-dying-how-do-you-design-queueing-timeouts-circuit-6f374640b55e), [FastAPI 502 + background](https://medium.com/@python-javascript-php-html-css/fixing-fastapi-background-task-502-error-on-aws-elastic-beanstalk-77e99ca2a453).

- **Схема:** клиент шлёт данные → API **мгновенно** возвращает 202 Accepted (или аналог) и **id задачи** → реальная обработка в воркере/очереди.
- **Плюсы:** нет долгого удержания соединения, нет 502 из-за таймаута обработки, можно масштабировать воркеры отдельно.

### 2.2. STT в продакшене: upload → событие → pipeline

Источники: [Speech pipeline orchestration](https://arunbaby.com/speech-tech/0031-speech-pipeline-orchestration/), [Whisper + AWS Batch](https://aws.amazon.com/blogs/hpc/whisper-audio-transcription-powered-by-aws-batch-and-aws-inferentia/), [Lambda + Whisper](https://towardsaws.com/building-a-speech-to-text-transcription-system-using-aws-lambda-and-openai-whisper-d670e85fa8df).

- Аудио кладут в **объектное хранилище** (S3, Blob).
- Загрузка файла **триггерит событие** (SQS, SNS, Lambda, Kafka).
- **Отдельный пайплайн** (Batch, K8s, Lambda): препроцессинг → VAD → ASR → постобработка. Каждый этап может быть своей очередью/сервисом.
- Ответ пользователю — не «дождаться транскрипции», а «файл принят, статус можно смотреть по id».

### 2.3. Клиент: очередь, ретраи, лимит нагрузки

Источники: обзоры по chunked upload, retry, backpressure (Flutter/upload best practices, resumable uploads).

- **Очередь загрузок** на устройстве, лимит **одновременных** запросов (часто 3–5).
- **Ретраи с exponential backoff + jitter**, ограничение числа попыток.
- **Сохранение прогресса** (какие чанки/файлы уже отправлены), чтобы после сбоя не слать всё заново.
- При потере сети — пауза, при появлении — возобновление.

### 2.4. Прокси: таймауты только как временная мера

Источники: [nginx proxy_read_timeout](https://nginx.org/r/proxy_read_timeout), [NGINX timeouts guide](https://www.getpagespeed.com/server-setup/nginx/nginx-timeout).

- **502/504** часто из-за того, что `proxy_read_timeout` (часто 60 сек) меньше времени ответа backend.
- **Правильный подход:** не «поднять таймаут до 10 минут», а **сделать ответ быстрым** (принял → 202 → обработка в воркере).
- Если всё же нужны долгие запросы — поднимать таймаут только для конкретного location и понимать риски (занятые соединения, память).

## 3. Что не так в текущей архитектуре Reflexio

| Аспект | Сейчас | Лучшая практика |
|--------|--------|------------------|
| **Момент ответа клиенту** | «received» и «transcription» после всей обработки (ASR + enrichment) | «received» сразу после приёма байтов; результат — позже (polling/webhook/отдельный канал) |
| **Где выполняется работа** | В том же процессе/соединении, что принял WebSocket | В отдельной очереди + воркеры (Celery, RQ, SQS, Lambda и т.д.) |
| **Длительность удержания соединения** | 30–120+ сек на один сегмент | &lt;1–2 сек на приём и ack |
| **Риск 502** | Высокий: прокси режет по таймауту | Низкий: быстрый ответ, тяжёлое — вне запроса |

Почему «у других нет»: у них **приём и обработка разделены**; у тебя пока **объединены в одном запросе**.

## 4. Что можно сделать (по приоритету)

### 4.1. Быстрый фикс (без смены архитектуры)

- **Увеличить таймауты прокси** для WebSocket и для long-polling, если есть:
  - nginx: `proxy_read_timeout`, `proxy_send_timeout` (например 300s для `/ws/ingest`).
  - Если перед backend ещё один proxy (cloud load balancer) — таймауты там тоже.
- **Плюс:** 502 может пропасть для сегментов, которые успевают обработаться за новый таймаут.
- **Минус:** под нагрузкой соединения будут висеть долго, память и лимиты соединений могут кончиться; это не масштабируемое решение.

### 4.2. Правильное направление: «принял → ack → обработал в фоне»

Цель: **не держать** WebSocket открытым на время ASR + enrichment.

Вариант A — **минимальное изменение протокола:**

1. В `websocket_ingest`: как только получили байты — **сразу** сохранить их в файл/хранилище, присвоить `file_id`, отправить клиенту `{"type": "received", "file_id": "..."}` и **не вызывать** `_process_audio_segment` в текущей корутине.
2. Поставить задачу в очередь (Celery, RQ, Redis list, или asyncio + отдельный процесс-воркер): «обработать file_id».
3. Воркер выполняет текущий `process_audio_bytes` (ASR, enrichment, БД).
4. Результат клиенту:
   - **вариант 1:** клиент периодически опрашивает `GET /ingest/status/{file_id}`;
   - **вариант 2:** при появлении результата сервер шлёт push (отдельный канал, webhook, или повторное подключение по WebSocket с подпиской по `file_id`).

Тогда соединение после «received» можно закрывать или оставлять только для следующих сегментов; таймаут прокси перестаёт упираться в длительность транскрипции.

Вариант B — **через объектное хранилище (как у крупных STT-систем):**

1. Клиент заливает WAV в S3/MinIO по presigned URL (или через один быстрый PUT на твой backend, который сразу кладёт файл в хранилище).
2. Backend только регистрирует «файл X принят», возвращает 202 и `job_id`.
3. Очередь (SQS, RabbitMQ, Redis) триггерит воркер; воркер качает файл, гонит через Whisper → enrichment → БД.
4. Статус/результат — через polling или webhook.

Это уже масштабируемая схема, но требует инфраструктуры (очередь, воркеры, хранилище).

### 4.3. Клиент (у тебя уже хорошо)

- Очередь на устройстве, повторные попытки, диагностика (полоска, 502, «Сервер перегружен…») — это как раз в духе лучших практик.
- Имеет смысл добавить/проверить:
  - **Ограничение одновременных загрузок** (например не больше 1–2 сегментов в полёте), чтобы не создавать десятки «долгих» соединений разом.
  - **Exponential backoff** при 502/5xx (если ещё не везде есть).

## 5. Итог одной фразой

**Почему у тебя проблемы, а у других нет:** у других тяжёлая обработка (STT, LLM) вынесена из запроса в очереди и воркеры, а ответ клиенту отдаётся сразу («принято»); у тебя пока вся обработка выполняется в том же соединении, которое держит открытым прокси → таймаут → 502 и рост очереди на телефоне.

**Что делать в первую очередь:** перейти к схеме «принял аудио → сразу ответил received → обработал в фоне (очередь + воркер) → результат через статус или push». Таймауты прокси тогда перестают быть причиной 502; при необходимости можно дополнительно поднять их как временную меру.
